## Resolved 3 agy review findings

### 1. [MEDIUM] Local Path Detection Heuristic Flaw — update-deps/SKILL.md
**Problem:** The `[ -d "$first_component" ]` check on the first path component (e.g., `mlx-community/`) would return true if a local directory happened to share the name of a HuggingFace org. This caused remote repo IDs like `mlx-community/whisper-large-v3-mlx` to be falsely classified as local paths when the org directory existed in the workspace, skipping remote update checks.

**Fix:** Wrapped the first-component directory check inside a guard that only activates when the URL starts with a local filesystem path indicator (`/`, `./`, or `../`). Bare repo IDs like `org/repo` now bypass the parent-directory heuristic entirely and proceed to the HF URL construction logic.

### 2. [MEDIUM] Port Configuration Disconnect — start-backends/SKILL.md
**Problem:** Overriding LLM_PORT, STT_PORT, or TTS_PORT caused the backend services to start on the custom ports, but the API Gateway's VLLM_URL, STT_URL, and TTS_URL still pointed to the default ports (8001, 8002, 8003), resulting in connection failures.

**Fix:** After setting the port variables, the script now exports matching URL variables with defaults that use the port values (e.g., `export VLLM_URL="${VLLM_URL:-http://127.0.0.1:$LLM_PORT}"`). Explicit env overrides of VLLM_URL/STT_URL/TTS_URL are still respected.

### 3. [LOW] Bash Sourcing Fragility for .env Files — both skills
**Problem:** `source .env` in bash mangles values with inline comments (e.g., `VAR=value # comment` sets VAR to "value # comment") and whitespace around `=`, differing from Python's python-dotenv behavior.

**Fix:** Replaced `set -a && source .env && set +a` with a Python-based parser using `shlex.quote()` that strips inline comments and trims whitespace. Applied identically to both start-backends/SKILL.md and update-deps/SKILL.md.

### Finding 4 (Low) — Already Resolved
The "Fragile Multi-line Python Execution via Double Quotes" finding was addressed in a previous commit (b418c5c) that replaced the double-quoted print statement inside the double-quoted bash string with single quotes. The current `uv run python -c "..."` block uses only single-quoted Python strings, so the quoting conflict was already eliminated.
